# 高校数据采集 MVP 执行计划

## 1. 背景与目标

当前需求已从「重做三栏投研 UI」调整为「基于现有 UI，优先建设高校数据采集底座」。本阶段不改造核心交互形态，不做新的投研工作台，先验证系统是否能持续、稳定地把高校公开信息转化为可用的结构化数据。

MVP 核心目标：

- 以清华大学为第一批试点高校，跑通高校公开数据采集闭环。
- 将院系、实验室、教师、学生团队、新闻动态、科研成果、竞赛获奖等公开网页结构化入库。
- 保留原始来源、正文、抓取时间和初步标签，支持后续线索研判。
- 复用现有 UI 展示采集结果，不新增复杂页面。
- 产出第一批「疑似项目线索」和「疑似人才线索」。

本阶段的产品定位是：高校数据采集底座版。

## 2. MVP 范围

### 2.1 数据源范围

第一批优先覆盖清华公开网页数据：

- 院系官网
- 实验室 / 课题组页面
- 教师主页 / 团队成员页面
- 新闻动态 / 科研进展页面
- 科研成果 / 成果转化页面
- 竞赛 / 获奖 / 项目发布页面
- 招生 / 招聘 / 项目合作页面

建议首批选择 5-10 个高价值来源，优先覆盖机器人、智能硬件、大模型、芯片、生物医药等方向相关院系或实验室。

### 2.2 数据处理范围

MVP 需完成以下处理：

- 手动或配置化维护高校数据源。
- 支持手动触发采集，后续补充定时采集。
- 抓取页面标题、正文、发布时间、URL、来源站点、抓取时间。
- 对采集结果做 URL 去重和内容指纹去重。
- 初步识别内容类型：人物、项目、论文、新闻、成果、赛事、招聘、其他。
- 初步识别对象：学校、院系、机构、人物、团队、项目、研究方向。
- 初步生成标签：赛道、技术方向、项目阶段、可信度、是否疑似线索。
- 支持人工确认、忽略、修改标签、标记重点关注。

### 2.3 展示范围

本阶段基于现有 UI：

- 复用现有列表展示采集结果。
- 复用现有详情页查看结构化字段、原文摘要和来源链接。
- 支持基础筛选：学校、院系、内容类型、赛道、是否疑似线索、是否已确认。
- 支持跳转原始来源 URL。

### 2.4 AI 能力范围

MVP 只做轻量抽取和分类，不做复杂投研推理：

- 判断是否高校相关。
- 判断是否人物线索。
- 判断是否项目线索。
- 判断是否成果转化 / 创业 / 商业化信号。
- 提取人物、机构、项目、论文、赛事、研究方向。
- 生成一段简短结构化摘要。
- 给出初步标签和置信度。

## 3. 暂缓范围

以下内容不进入 MVP：

- 新三栏 UI / ChatGPT 式投研工作台。
- 单项目深度对话式研判。
- 每日 Top5 正式投研报告。
- 多模型协同调度。
- 用户习惯自学习。
- 微博、B 站、小红书、抖音、LinkedIn 等社交媒体采集。
- 700+ 公众号全量监测。
- 万人级人才库扩容。
- 人物关系图谱。
- 创业概率评分。
- Alpha 信号成熟规则闭环。
- 底层存储性能专项优化。

## 4. 阶段排期

整体周期建议为 3-4 周。

| 阶段 | 时间 | 目标 | 主要交付物 |
| --- | --- | --- | --- |
| 第 0 阶段 | 1-2 天 | 明确 MVP 边界和字段 | 数据源清单、字段定义、采集任务类型、去重规则 |
| 第 1 阶段 | 3-5 天 | 跑通清华网页采集 | 院系 / 实验室 / 新闻 / 教师页采集器，原文 URL 与正文入库 |
| 第 2 阶段 | 3-5 天 | 完成结构化抽取和分类 | 人物、项目、论文、成果、赛事、新闻等类型识别，初步标签 |
| 第 3 阶段 | 2-4 天 | 接入现有 UI 查看 | 列表、详情、来源跳转、基础筛选 |
| 第 4 阶段 | 3-5 天 | 完成去重、更新和任务管理 | URL 去重、内容指纹去重、重复采集更新、手动触发采集 |
| 第 5 阶段 | 2-4 天 | 完成最小人工校正 | 人工确认 / 忽略线索、修改标签、标记重点关注 |
| 第 6 阶段 | 3-5 天 | 形成第一版高校线索池 | 疑似项目线索列表、疑似人才线索列表、采集质量报告 |

## 5. 推荐里程碑

### 第 1 周：清华网页采集闭环

目标：

- 完成首批数据源清单。
- 跑通 5-10 个清华公开来源采集。
- 原始 HTML / 正文 / URL / 抓取时间可入库。
- 初步完成 URL 去重。

验收：

- 任意一个来源可手动触发采集。
- 重复采集不会重复入库。
- 每条记录都能追溯原始 URL。

### 第 2 周：结构化抽取与标签

目标：

- 从正文中提取人物、机构、项目、研究方向。
- 识别内容类型。
- 输出初步赛道标签和疑似线索判断。
- 完成内容指纹去重。

验收：

- 至少能区分人物、项目、新闻、论文、成果、赛事。
- 每条记录有结构化摘要。
- 可筛出一批疑似项目线索。

### 第 3 周：接入现有 UI 与人工校正

目标：

- 在现有 UI 中查看采集数据。
- 支持查看详情和来源跳转。
- 支持人工确认、忽略、改标签。

验收：

- 投研人员能在现有页面完成查看、筛选、确认、忽略。
- 人工处理结果能保存并影响后续列表状态。

### 第 4 周：扩源与稳定性

目标：

- 扩展更多院系 / 实验室来源。
- 补充采集日志、失败重试、异常记录。
- 形成第一批高校线索池。

验收：

- 可稳定采集 10 个以上来源。
- 失败任务可定位原因。
- 输出第一版疑似高校项目线索池。

## 6. 数据字段建议

### 6.1 原始采集字段

- source_id：数据源 ID
- source_name：数据源名称
- source_type：数据源类型
- school：学校
- department：院系 / 机构
- url：原始 URL
- title：页面标题
- published_at：发布时间
- crawled_at：抓取时间
- raw_html：原始 HTML
- extracted_text：正文
- content_hash：内容指纹
- status：采集状态

### 6.2 结构化字段

- content_type：内容类型
- people：人物列表
- organizations：机构 / 实验室 / 团队列表
- projects：项目 / 成果列表
- papers：论文列表
- competitions：赛事 / 奖项列表
- research_directions：研究方向
- technologies：技术关键词
- tracks：赛道标签
- summary：结构化摘要
- confidence：抽取置信度

### 6.3 线索判断字段

- is_potential_lead：是否疑似线索
- lead_type：人物线索 / 项目线索 / 技术成果线索 / 赛事线索 / 其他
- commercialization_signal：是否存在商业化 / 成果转化信号
- startup_signal：是否存在创业相关信号
- priority：优先级
- review_status：未处理 / 已确认 / 已忽略 / 待补充
- reviewer_note：人工备注

## 7. P0 / P1 / P2 优先级

### P0 必做

- 高校数据源配置。
- 网页采集与正文抽取。
- 原始数据入库。
- URL 去重。
- 内容指纹去重。
- 人物 / 项目 / 论文 / 新闻 / 成果分类。
- 来源链接保留。
- 现有 UI 查看。
- 人工确认 / 忽略。

### P1 可跟进

- 定时采集。
- 标签体系完善。
- 多院系扩展。
- 采集失败重试。
- 简单采集日志。
- 疑似创业 / 成果转化信号识别。
- 批量导入数据源。

### P2 暂缓

- 三栏新 UI。
- 投研对话工作台。
- 每日 Top5 报告。
- 多模型协同。
- 社交媒体采集。
- 人物关系图谱。
- 创业概率评分。

## 8. 验收标准

MVP 完成后，应能跑通以下链路：

配置高校 / 院系数据源 -> 手动或定时采集 -> 页面解析 -> 结构化入库 -> 去重 -> 初步标签 -> 现有 UI 查看 -> 人工确认 / 忽略 -> 形成疑似线索池

最低验收标准：

- 清华至少接入 5-10 个院系 / 实验室来源。
- 每天可重复采集，不重复入库。
- 每条记录保留原始 URL、正文和抓取时间。
- 能识别人物、项目、论文、新闻、成果、赛事几类内容。
- 能筛出一批疑似项目线索。
- 投研人员可以在现有 UI 中查看、筛选、确认和忽略。

## 9. 风险与依赖

### 9.1 主要风险

- 高校网页结构不统一，采集器需要兼容多种页面。
- 部分页面缺少发布时间，影响时间排序。
- 页面正文抽取质量不稳定，可能混入导航、页脚、无关文本。
- AI 抽取结果可能误判，需要人工校正闭环。
- 数据源扩展过快会拖慢 MVP 验证。

### 9.2 应对策略

- 第一阶段只选高质量、结构稳定的数据源。
- 正文抽取先用通用规则，遇到高价值站点再做专属适配。
- 对发布时间缺失的数据标记为待补充，不阻塞入库。
- 所有 AI 判断保留置信度和人工状态。
- MVP 只追求 5-10 个来源稳定跑通，不追求全量覆盖。
